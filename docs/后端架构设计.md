## 一、架构总览

### 核心原则

1. **Supabase 自部署**为数据层核心（PostgreSQL + Auth + Realtime + REST API）
2. **Hono Worker** 处理支付回调和数据采集（轻量 Node.js 服务）
3. **静态支付页**独立部署（纯 HTML，无框架开销）
4. **Chrome Extension** 通过 Supabase JS SDK 直连数据库（省去中间 API 层）

### 为什么这样分层

| 层 | 技术 | 理由 |
|---|---|---|
| **数据库 + API + 认证** | Supabase 自部署 | 自动生成 REST API（扩展端直接查询）；内置手机号登录；Realtime 推送替代轮询 |
| **数据采集 + 支付回调** | Hono Worker | 需要定时任务 + HTTP 客户端 + 支付签名 — 不适合 Edge Function |
| **支付网页** | 纯静态 HTML | 3 个页面，CDN 部署最快最便宜 |
| **反向代理** | Nginx | HTTPS 终端 + 路由分发 |

### 对比原方案（Fastify + Prisma）

| 维度 | Fastify + Prisma | Supabase + Hono Worker |
|------|-----------------|----------------------|
| API 路由代码 | 手写 ~30 个路由 | PostgREST 自动生成，零代码 |
| 认证 | 手写 JWT + SMS | Supabase Auth 内置 |
| 实时推送 | 轮询或自建 WebSocket | Supabase Realtime 内置 |
| 扩展端 SDK | 手写 apiClient | @supabase/supabase-js 官方 SDK |
| 数据采集代码 | 同 | 同（Hono 处理） |
| 预估后端代码量 | ~3000 行 | ~1000 行（仅 Worker） |

**Row Level Security (RLS)**：

以下策略在**第 3 节建表完成后**执行：

```sql
-- 设计原则：
-- 1. 前端只直连“本人数据”和“公开税率只读数据”
-- 2. 所有会影响余额 / 套餐 / 订单状态的写操作，一律走 RPC 或 Worker
-- 3. service_role 仅保留在 Worker，扩展端永不持有

-- ==================== 用户资料 ====================
ALTER TABLE public.users ENABLE ROW LEVEL SECURITY;

CREATE POLICY "users_self_select"
  ON public.users
  FOR SELECT TO authenticated
  USING (id = auth.uid());

-- public.users 不给客户端开放直接 UPDATE
-- 如需修改 display_name / email 等白名单字段，建议单独提供 update_user_profile RPC

-- ==================== 订单 / 权益 ====================
ALTER TABLE public.payment_orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY "orders_self_select"
  ON public.payment_orders
  FOR SELECT TO authenticated
  USING (user_id = auth.uid());

ALTER TABLE public.credit_transactions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "credits_self_select"
  ON public.credit_transactions
  FOR SELECT TO authenticated
  USING (user_id = auth.uid());

ALTER TABLE public.plan_subscriptions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "plans_self_select"
  ON public.plan_subscriptions
  FOR SELECT TO authenticated
  USING (user_id = auth.uid());

-- ==================== 订阅 / 通知配置 ====================
ALTER TABLE public.subscriptions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "subs_self_crud"
  ON public.subscriptions
  FOR ALL TO authenticated
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

ALTER TABLE public.notification_config ENABLE ROW LEVEL SECURITY;

CREATE POLICY "notify_config_self_crud"
  ON public.notification_config
  FOR ALL TO authenticated
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

ALTER TABLE public.user_notification_events ENABLE ROW LEVEL SECURITY;

CREATE POLICY "notify_events_self_select"
  ON public.user_notification_events
  FOR SELECT TO authenticated
  USING (user_id = auth.uid());

CREATE POLICY "notify_events_self_update"
  ON public.user_notification_events
  FOR UPDATE TO authenticated
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

-- ==================== 公开税率只读数据 ====================
ALTER TABLE public.tariff_dict ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.tariff_sub_codes ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.section_301_rates ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.section_232_rates ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.ad_cvd_orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.fta_rates ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.shipping_routes ENABLE ROW LEVEL SECURITY;

CREATE POLICY "tariff_dict_public_read"
  ON public.tariff_dict
  FOR SELECT TO authenticated, anon
  USING (true);

CREATE POLICY "tariff_sub_codes_public_read"
  ON public.tariff_sub_codes
  FOR SELECT TO authenticated, anon
  USING (true);

CREATE POLICY "section_301_public_read"
  ON public.section_301_rates
  FOR SELECT TO authenticated, anon
  USING (true);

CREATE POLICY "section_232_public_read"
  ON public.section_232_rates
  FOR SELECT TO authenticated, anon
  USING (true);

CREATE POLICY "ad_cvd_public_read"
  ON public.ad_cvd_orders
  FOR SELECT TO authenticated, anon
  USING (true);

CREATE POLICY "fta_public_read"
  ON public.fta_rates
  FOR SELECT TO authenticated, anon
  USING (true);

CREATE POLICY "shipping_public_read"
  ON public.shipping_routes
  FOR SELECT TO authenticated, anon
  USING (true);

-- tariff_change_log 仅供 Worker 内部使用，不给客户端任何 policy
ALTER TABLE public.tariff_change_log ENABLE ROW LEVEL SECURITY;

-- 写路径约束：
-- - authenticated: create_payment_order / create_subscription / update_notification_config
-- - service_role : fulfill_payment_order / collect_* / push_* / admin_*
-- - anon/public   : 不授予任何敏感 RPC 执行权限
```

> 直连 Supabase 不等于“前端直接改表”。推荐模式是：读操作走表 / 视图 / 只读 RPC，涉及订单、积分、套餐、签到、通知投递状态的写操作统一走 RPC。

### 2.2 Hono Worker（Node.js）

```typescript
// src/app.ts
import { Hono } from 'hono'
import { cors } from 'hono/cors'
import { logger } from 'hono/logger'
import { createClient } from '@supabase/supabase-js'

const app = new Hono()

// 中间件
app.use('*', logger())
app.use('*', cors({ origin: ['chrome-extension://*', 'https://pay.tarifftool.com'] }))

// Supabase Admin Client（service_role，绕过 RLS）
const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
)

// 路由注册
app.route('/payments', paymentRoutes)    // 支付回调
app.route('/collect', collectRoutes)     // 手动触发采集
app.route('/notify', notifyRoutes)       // 通知推送
app.route('/admin', adminRoutes)         // 管理接口

// 启动定时任务
startCronJobs()

export default app
```

## 三、数据库设计（完整版）

### 3.1 用户 & 支付

```sql
-- Supabase Auth 自动管理 auth.users（手机号、密码、JWT）
-- public.users 是扩展表，通过 trigger 与 auth.users 同步

CREATE TABLE public.users (
  id              UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  phone           VARCHAR(20) UNIQUE,
  email           VARCHAR(255),
  display_name    VARCHAR(50),

  -- 积分
  credits         INTEGER DEFAULT 100,
  total_earned    INTEGER DEFAULT 100,
  total_spent     INTEGER DEFAULT 0,

  -- 套餐
  plan            VARCHAR(20) DEFAULT 'free',
  plan_expires_at TIMESTAMPTZ,
  trial_ends_at   TIMESTAMPTZ,

  -- 签到
  checkin_streak  INTEGER DEFAULT 0,
  last_checkin_at TIMESTAMPTZ,

  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- auth.users 创建时自动创建 public.users
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.users (id, phone, email)
  VALUES (NEW.id, NEW.phone, NEW.email);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

-- 支付订单
CREATE TABLE public.payment_orders (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES public.users(id),
  order_no        VARCHAR(32) UNIQUE NOT NULL,
  order_type      VARCHAR(20) NOT NULL,           -- credits / plan
  status          VARCHAR(20) DEFAULT 'pending',   -- pending/paid/failed/expired/refunded

  amount_cents    INTEGER NOT NULL,                -- 金额（分）
  currency        VARCHAR(3) DEFAULT 'CNY',

  credit_amount   INTEGER,                         -- 积分数
  plan_tier       VARCHAR(20),                     -- 套餐等级
  plan_cycle      VARCHAR(10),                     -- monthly / yearly

  payment_channel VARCHAR(20),                     -- wechat / alipay / stripe
  payment_no      VARCHAR(64),                     -- 三方流水号

  paid_at         TIMESTAMPTZ,
  expired_at      TIMESTAMPTZ DEFAULT (NOW() + INTERVAL '30 minutes'),
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_orders_user ON payment_orders(user_id);
CREATE INDEX idx_orders_status ON payment_orders(status) WHERE status = 'pending';

-- 积分流水
CREATE TABLE public.credit_transactions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES public.users(id),
  order_id        UUID REFERENCES public.payment_orders(id),
  type            VARCHAR(10) NOT NULL,            -- earn / spend
  amount          INTEGER NOT NULL,
  reason          VARCHAR(50) NOT NULL,
  balance_after   INTEGER NOT NULL,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_credits_user ON credit_transactions(user_id);
CREATE UNIQUE INDEX idx_credit_purchase_order
  ON public.credit_transactions(order_id)
  WHERE order_id IS NOT NULL AND reason = 'purchase';

-- 套餐订阅
CREATE TABLE public.plan_subscriptions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES public.users(id),
  order_id        UUID REFERENCES public.payment_orders(id),
  plan_tier       VARCHAR(20) NOT NULL,
  cycle           VARCHAR(10) NOT NULL,
  status          VARCHAR(20) DEFAULT 'active',
  started_at      TIMESTAMPTZ DEFAULT NOW(),
  expires_at      TIMESTAMPTZ NOT NULL,
  cancelled_at    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE UNIQUE INDEX idx_plan_order
  ON public.plan_subscriptions(order_id)
  WHERE order_id IS NOT NULL;

CREATE UNIQUE INDEX idx_plan_active_user
  ON public.plan_subscriptions(user_id)
  WHERE status = 'active';
```

```sql
CREATE TABLE public.tariff_dict (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  country_code VARCHAR(2) NOT NULL,              -- "US" / "EU" / "JP"
  hs6          VARCHAR(6) NOT NULL,              -- "848210"
  name_zh      VARCHAR(200),
  name_en      VARCHAR(200),
  keywords     TEXT[],                           -- {"轴承","bearing","滚珠"}
  created_at   TIMESTAMPTZ DEFAULT NOW(),
  updated_at   TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(country_code, hs6)

-- 目标国 HTS 子分类
CREATE TABLE public.tariff_sub_codes (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  code            VARCHAR(20) NOT NULL,           -- "8482.10.50.00"
  description_zh  VARCHAR(500),
  description_en  VARCHAR(500),
  mfn_rate        DECIMAL(6,4) DEFAULT 0,
  unit            VARCHAR(10),
  special_rate    DECIMAL(6,4),
  effective_date  DATE,
  source          VARCHAR(20),                   -- "hts"/"taric"/"wits"
```

### 4.5 数据标准化

```typescript
// services/collect/normalizer.ts

/** HS 编码标准化：去非数字，取前 N 位 */
export function normalizeHS(raw: string, digits: 6 | 8 | 10 = 6): string {
  return raw.replace(/[^0-9]/g, '').substring(0, digits)
}

/** 税率标准化：百分号字符串 → 小数 */
export function normalizeRate(raw: string | number): number {
  if (typeof raw === 'number') return raw
  if (raw === 'Free' || raw === '0') return 0
  const num = parseFloat(raw.replace('%', ''))
  return num > 1 ? num / 100 : num  // "9%" → 0.09, "0.09" → 0.09
}

/** 确保字典 HS6 条目存在，不存在则创建 */
export async function getOrCreateDictId(
  hs6: string,
  countryCode: string
): Promise<string> {
  const normalized = normalizeHS(hs6)

  const { data } = await supabase
    .from('tariff_dict')
    .select('id')
    .eq('country_code', countryCode)
    .eq('hs6', normalized)
    .maybeSingle()

  if (data) return data.id

  const { data: created } = await supabase
    .from('tariff_dict')
    .insert({
      country_code: countryCode,
      hs6: normalized,
      name_en: `HS ${normalized}`,
      keywords: [normalized],
    })
    .select('id')
    .single()

  return created!.id
}
```

### 5.2 创建订单（扩展端通过 Supabase RPC 调用）

```sql
-- Supabase PostgreSQL Function
-- 扩展端调用: supabase.rpc('create_payment_order', { ... })
CREATE OR REPLACE FUNCTION public.create_payment_order(
  p_order_type VARCHAR,
  p_payment_channel VARCHAR,
  p_credit_pack_id VARCHAR DEFAULT NULL,
  p_plan_tier VARCHAR DEFAULT NULL,
  p_plan_cycle VARCHAR DEFAULT NULL
)
RETURNS JSON
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_user_id UUID := auth.uid();
  v_order_no VARCHAR(32);
  v_amount_cents INTEGER;
  v_credit_amount INTEGER;
  v_order_id UUID;
  v_payment_page_url VARCHAR;
BEGIN
  IF v_user_id IS NULL THEN
    RAISE EXCEPTION 'not_authenticated';
  END IF;

  IF p_payment_channel NOT IN ('wechat', 'alipay') THEN
    RAISE EXCEPTION 'unsupported_payment_channel';
  END IF;

  -- 生成订单号
  v_order_no := 'TFP' || to_char(NOW(), 'YYYYMMDDHH24MISS') || lpad(floor(random() * 10000)::text, 4, '0');

  -- 计算金额
  IF p_order_type = 'credits' THEN
    v_amount_cents := CASE p_credit_pack_id
      WHEN 'small'   THEN 900
      WHEN 'medium'  THEN 3900
      WHEN 'large'   THEN 9900
      WHEN 'xlarge'  THEN 28900
      WHEN 'xxlarge' THEN 99900
    END;
    v_credit_amount := CASE p_credit_pack_id
      WHEN 'small'   THEN 100
      WHEN 'medium'  THEN 500
      WHEN 'large'   THEN 1500
      WHEN 'xlarge'  THEN 5000
      WHEN 'xxlarge' THEN 20000
    END;
    IF v_amount_cents IS NULL OR v_credit_amount IS NULL THEN
      RAISE EXCEPTION 'invalid_credit_pack';
    END IF;
  ELSIF p_order_type = 'plan' THEN
    v_amount_cents := CASE p_plan_tier || '-' || p_plan_cycle
      WHEN 'starter-monthly'  THEN 2900
      WHEN 'starter-yearly'   THEN 29000
      WHEN 'pro-monthly'      THEN 7900
      WHEN 'pro-yearly'       THEN 69900
      WHEN 'enterprise-monthly' THEN 29900
      WHEN 'enterprise-yearly'  THEN 299900
    END;
    IF v_amount_cents IS NULL THEN
      RAISE EXCEPTION 'invalid_plan';
    END IF;
  ELSE
    RAISE EXCEPTION 'invalid_order_type';
  END IF;

  -- 创建订单
  INSERT INTO payment_orders (
    user_id, order_no, order_type, status,
    amount_cents, credit_amount, plan_tier, plan_cycle,
    payment_channel
  ) VALUES (
    v_user_id, v_order_no, p_order_type, 'pending',
    v_amount_cents, v_credit_amount, p_plan_tier, p_plan_cycle,
    p_payment_channel
  ) RETURNING id INTO v_order_id;

  -- 生成支付页 URL
  v_payment_page_url := 'https://pay.tarifftool.com/' || v_order_id || '?channel=' || p_payment_channel;

  RETURN json_build_object(
    'orderId', v_order_id,
    'orderNo', v_order_no,
    'amountYuan', v_amount_cents / 100.0,
    'paymentPageUrl', v_payment_page_url,
    'expireAt', NOW() + INTERVAL '30 minutes'
  );
END;
$$;

REVOKE ALL ON FUNCTION public.create_payment_order(VARCHAR, VARCHAR, VARCHAR, VARCHAR, VARCHAR) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.create_payment_order(VARCHAR, VARCHAR, VARCHAR, VARCHAR, VARCHAR) TO authenticated;
```

> 关键约束：
> 1. 建单函数内部始终使用 `auth.uid()`，不接受客户端传入 `user_id`。
> 2. 扩展端只负责发起建单，不允许直接写 `payment_orders.status`、`users.credits`、`plan_subscriptions`。
> 3. 所有支付成功后的履约操作都由 Worker 调用 `fulfill_payment_order` 完成。

### 5.3 订单履行（支付成功后）

```sql
-- 由 Worker 在支付回调验签成功后调用
-- 目标：单事务内完成“锁单 → 改状态 → 发放权益 → 记录流水”
CREATE OR REPLACE FUNCTION public.fulfill_payment_order(
  p_order_id UUID,
  p_payment_channel VARCHAR,
  p_payment_no VARCHAR,
  p_paid_at TIMESTAMPTZ DEFAULT NOW()
)
RETURNS JSON
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_order public.payment_orders%ROWTYPE;
  v_balance_after INTEGER;
  v_base_expires_at TIMESTAMPTZ;
  v_expires_at TIMESTAMPTZ;
BEGIN
  SELECT *
    INTO v_order
  FROM public.payment_orders
  WHERE id = p_order_id
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'order_not_found';
  END IF;

  IF v_order.status = 'paid' THEN
    RETURN json_build_object(
      'ok', true,
      'idempotent', true,
      'orderId', v_order.id
    );
  END IF;

  IF v_order.status <> 'pending' THEN
    RAISE EXCEPTION 'order_status_invalid';
  END IF;

  UPDATE public.payment_orders
  SET status = 'paid',
      payment_channel = p_payment_channel,
      payment_no = p_payment_no,
      paid_at = p_paid_at
  WHERE id = v_order.id;

  IF v_order.order_type = 'credits' THEN
    UPDATE public.users
    SET credits = credits + v_order.credit_amount,
        total_earned = total_earned + v_order.credit_amount,
        updated_at = NOW()
    WHERE id = v_order.user_id
    RETURNING credits INTO v_balance_after;

    INSERT INTO public.credit_transactions (
      user_id, order_id, type, amount, reason, balance_after
    ) VALUES (
      v_order.user_id, v_order.id, 'earn', v_order.credit_amount, 'purchase', v_balance_after
    )
    ON CONFLICT DO NOTHING;
  END IF;

  IF v_order.order_type = 'plan' THEN
    SELECT COALESCE(MAX(expires_at), p_paid_at)
      INTO v_base_expires_at
    FROM public.plan_subscriptions
    WHERE user_id = v_order.user_id
      AND status = 'active'
      AND expires_at > p_paid_at;

    v_expires_at := v_base_expires_at +
      CASE WHEN v_order.plan_cycle = 'yearly'
        THEN INTERVAL '1 year'
        ELSE INTERVAL '1 month'
      END;

    UPDATE public.plan_subscriptions
    SET status = 'cancelled',
        cancelled_at = p_paid_at
    WHERE user_id = v_order.user_id
      AND status = 'active';

    INSERT INTO public.plan_subscriptions (
      user_id, order_id, plan_tier, cycle, status, started_at, expires_at
    ) VALUES (
      v_order.user_id, v_order.id, v_order.plan_tier, v_order.plan_cycle, 'active', p_paid_at, v_expires_at
    );

    UPDATE public.users
    SET plan = v_order.plan_tier,
        plan_expires_at = v_expires_at,
        updated_at = NOW()
    WHERE id = v_order.user_id;
  END IF;

  RETURN json_build_object(
    'ok', true,
    'idempotent', false,
    'orderId', v_order.id
  );
END;
$$;

REVOKE ALL ON FUNCTION public.fulfill_payment_order(UUID, VARCHAR, VARCHAR, TIMESTAMPTZ) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.fulfill_payment_order(UUID, VARCHAR, VARCHAR, TIMESTAMPTZ) TO service_role;
```

```typescript
// services/notify/index.ts
export async function pushChangeToSubscribers(change: TariffChangeLog) {
  // 查找匹配的订阅规则
  const { data: subs } = await supabase
    .from('subscriptions')
    .select('*, users(*)')
    .eq('is_active', true)

  for (const sub of subs ?? []) {
    if (!matchesRule(sub.rule_config, change)) continue

    const config = await getNotificationConfig(sub.user_id)

    // 按渠道推送
    for (const channel of sub.channels) {
      const dedupeKey = `${change.id}:${sub.user_id}:${channel}`

      const { data: event } = await supabase
        .from('user_notification_events')
        .upsert({
          user_id: sub.user_id,
          change_id: change.id,
          subscription_id: sub.id,
          channel,
          title: buildNotificationTitle(change),
          payload: {
            hsCode: change.hs_code,
            originCountry: change.origin_country,
            destinationCountry: change.destination_country,
            changeType: change.change_type,
            oldRate: change.old_rate,
            newRate: change.new_rate,
            effectiveDate: change.effective_date,
            source: change.source,
          },
          dedupe_key: dedupeKey,
        }, { onConflict: 'dedupe_key' })
        .select('*')
        .single()

      switch (channel) {
        case 'email':
          if (config.channels.email.enabled) {
            await sendEmail(config.channels.email.address, change)
            await markDelivered(event.id)
          }
          break
        case 'dingtalk':
          if (config.channels.dingtalk.enabled) {
            await sendDingTalk(config.channels.dingtalk, change)
            await markDelivered(event.id)
          }
          break
        case 'wechat':
          if (config.channels.wechat.enabled) {
            await sendWeChat(config.channels.wechat, change)
            await markDelivered(event.id)
          }
          break
        case 'webhook':
          if (config.channels.webhook.enabled) {
            await sendWebhook(config.channels.webhook.url, change)
            await markDelivered(event.id)
          }
          break
        case 'system':
          // system 渠道不在服务端主动发送
          // 只写入 user_notification_events，由扩展端通过 Realtime 订阅自己的事件
          break
      }
    }
  }
}
```

其中 `buildNotificationTitle()` 用于生成通知标题，`markDelivered(eventId)` 负责把 `user_notification_events.status` 更新为 `delivered` 并写入 `delivered_at`。

### 6.3 Chrome Extension 系统通知

通过 Supabase Realtime 订阅自己的 `user_notification_events`，而不是订阅全局 `tariff_change_log`：

```typescript
// Extension 端监听“我的系统通知”
supabase
  .channel(`notify-${user.id}`)
  .on('postgres_changes', {
    event: 'INSERT',
    schema: 'public',
    table: 'user_notification_events',
    filter: `user_id=eq.${user.id}`,
  }, (payload) => {
    const event = payload.new
    if (event.channel !== 'system') return

    const change = event.payload
    chrome.notifications.create({
      type: 'basic',
      iconUrl: 'icon-128.png',
      title: '税率变化提醒',
      message: `${change.changeType}: ${change.hsCode} (${change.originCountry}→${change.destinationCountry}) 税率从 ${(change.oldRate * 100).toFixed(1)}% 变为 ${(change.newRate * 100).toFixed(1)}%`,
    })

    supabase
      .from('user_notification_events')
      .update({
        status: 'delivered',
        delivered_at: new Date().toISOString(),
      })
      .eq('id', event.id)
  })
  .subscribe()
```

## 七、扩展端改动总结

### 替换现有 apiClient → Supabase SDK

| 现有 | 替换为 |
|------|--------|
| `services/api-client.ts` | 新增 `services/supabase.ts` 适配层，保留少量 Worker HTTP 调用 |
| `services/auth.ts`（手写手机号登录） | `supabase.auth.signInWithOtp({ phone })`，保留 guest / offline fallback |
| `services/credits.ts` | `supabase.from('credit_transactions')` 只读 + 积分变更 RPC |
| `services/plans.ts` | `supabase.from('plan_subscriptions')` 只读 + 订阅建单 RPC |
| `store/*.ts` 持久化 | 继续使用 `chrome.storage` 作离线缓存，登录后按需回源 |
| 支付轮询 | `payment_orders` Realtime 订阅（仅本人订单） |

### 迁移原则

1. **保留离线优先**：策略输入、上次结果、HS 字典缓存、通知已读状态仍保留在本地存储中。
2. **读写分离**：前端直连 Supabase 主要用于读取本人数据与公开税率；订单、积分、套餐、签到等写操作统一走 RPC。
3. **保留映射层**：新增 snake_case → camelCase 的 adapter，避免数据库字段风格直接渗透到 React / Zustand。

### 新增数据流

```
扩展端打开 → 查询 Supabase tariff_dict（替代 mock-data）
           → 用户选择 HS 子分类 → 本地策略计算引擎（不变）
           → 点击购买 → Supabase RPC 创建订单 → 打开支付页
           → Realtime 订阅订单状态 → 支付成功 → 刷新本地状态
```
