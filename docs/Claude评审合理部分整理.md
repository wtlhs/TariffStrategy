# Claude 评审合理部分整理

> 来源：`claude评审对话.txt`
>
> 整理原则：仅保留**当前代码库中仍成立**、且对架构收敛、上线准备或后续实施仍有指导价值的评审结论。
>
> 不纳入本文件的内容：
> 1. 已被后续代码修复的判断
> 2. 明显基于旧快照、与当前仓库状态不一致的判断
> 3. 仅属表达层面的泛泛建议、缺少明确落点的内容

---

## 一、总体结论

Claude 评审里最值得保留的总判断是：

1. 项目整体技术路线是合理的，前后端分层、文档体系和商业化抽象都具备不错基础。
2. 当前项目更接近“工程化较好的 MVP / 可演示版本”，但距离“安全上线、可信收费”仍有几条关键链路需要补齐。
3. 最重要的不是重做技术选型，而是把**真实数据链路、后端权威化权益校验、生产级配置与测试**补完整。

这个总体结论与当前仓库状态基本一致。

---

## 二、仍然成立的核心评审点

### 1. 真实数据链路尚未完全闭环

这条是 Claude 评审中最有价值的结论之一，当前仍然成立。

现状：

1. `src/lib/strategy-engine.ts` 仍直接依赖 `mock-data.ts` 中的 `DEMO_ORIGINS`、税费映射和演示数据。
2. `src/services/tariff-dict.ts` 虽然已经具备“远程拉取 + 本地缓存 + mock 回退”的结构，但策略计算引擎本身仍不是纯粹依赖真实后端数据。
3. `src/entrypoints/background.ts` 的税率变化检查仍调用 `detectSimulatedChanges()`，没有真正消费后端推送或后端通知事件。

保留这条建议的原因：

1. 产品核心价值是“真实税率驱动的策略对比”，只要引擎仍大量依赖 mock 数据，商业可信度就会受限。
2. 后端采集器、Supabase 表、通知链路都已经在建设，如果前端主路径继续走演示数据，会造成“后端真数据”和“前端展示结果”长期割裂。

建议保留为 P0/P1 级事项：

1. 收敛策略引擎的数据输入层，只让 engine 消费“规范化后的真实数据模型”。
2. 把模拟通知和真实通知彻底拆开，生产环境只允许后端 fan-out 的事件进入用户通知流。
3. 明确哪些字段仍允许本地 fallback，哪些字段必须以后端为准。

---

### 2. 客户端不应作为权益与计费的最终权威

这条判断当前仍然非常准确。

现状：

1. `src/services/credits.ts` 中的签到、补签、积分消耗仍是“先改本地 store，再异步请求后端”。
2. `src/services/plans.ts` 中 `startTrial()` 仍会在前端请求成功后直接写入本地套餐状态。
3. 当前本地状态仍承载了大量“能不能用某功能”的业务判断。

风险：

1. 本地 `chrome.storage` / Zustand 状态可被篡改，导致白嫖、试用重置、前后端余额漂移。
2. 即使后端后续补上收费能力，如果前端继续把本地状态当权威，也会出现 UI 与真实权限不一致的问题。

建议保留为 P0 级事项：

1. 所有积分扣减、套餐升级、试用发放、付费功能解锁都以后端返回结果为准。
2. 前端最多做“乐观显示”，失败时必须回滚。
3. 对 AI 分析、深度报告、订阅创建等核心收费动作，统一改为后端验权后下发结果或短期凭证。

---

### 3. RLS 与权限边界必须系统验证，而不是只“写出来”

这条建议方向完全正确，而且对当前项目依然有价值。

现状：

1. 当前 `backend/supabase/migrations/20260424000001_init_schema.sql` 已经补上了较完整的 RLS 策略与 RPC 授权边界。
2. 但“策略已存在”不等于“越权路径已被验证”，尤其是在浏览器扩展直接持有 `Supabase URL + anon key` 的前提下。

为什么仍需保留：

1. RLS 是这类“前端直连 Supabase”架构的最后一道安全边界。
2. 一旦某张表策略漏写、某个 RPC 授权过宽，匿名或普通登录用户就可能直接越权访问敏感数据。

建议保留为上线前必做清单：

1. 逐表验证 `anon / authenticated / service_role` 三种身份的实际权限。
2. 重点验证 `users`、`payment_orders`、`credit_transactions`、`plan_subscriptions`、`subscriptions`、`notification_config`、`user_notification_events`。
3. 重点验证 `create_payment_order`、`fulfill_payment_order`、`create_subscription`、`update_notification_config` 等 RPC 的授权边界。

---

### 4. `options/App.tsx` 过重，应该拆分

这条建议当前依然成立，而且证据很明确。

现状：

1. `src/entrypoints/options/App.tsx` 当前仍有 700+ 行。
2. 文件同时承担了导航、主题初始化、Badge 清理、页面切换以及 `StrategyPage` 的整页实现。

为什么值得保留：

1. 这是典型的 P1 技术债，不一定阻塞上线，但会显著提高后续改动的认知成本和回归风险。
2. 当前文件结构已经偏离“入口只做装配，页面各自独立”的理想状态。

建议保留的重构方向：

1. 把 `StrategyPage` 单独拆到 `src/entrypoints/options/pages/StrategyPage.tsx`。
2. 再细分成搜索表单、HS 选择器、KPI 卡片、结果表格、说明弹窗等子组件。
3. `App.tsx` 尽量只保留 layout、导航和页面路由装配。

---

### 5. 模拟通知逻辑必须加开发环境隔离

这条建议当前仍成立，而且是非常低成本的修复项。

现状：

1. `src/services/notification-service.ts` 中 `detectSimulatedChanges()` 仍会随机生成模拟税率变化。
2. `src/entrypoints/background.ts` 中 `handleTariffCheck()` 仍直接调用它。

风险：

1. 只要这条逻辑进入生产，用户就可能收到基于假数据的通知。
2. 这不是普通 bug，而是会直接损害产品可信度的问题。

建议保留为 P1 级小修复：

1. 用 `import.meta.env.DEV` 或等价机制把模拟逻辑限制在开发环境。
2. 更稳妥的做法是把模拟通知代码与正式通知代码完全隔离，避免误触发。

---

### 6. 采集器的稳定性与数据质量监控不足

这条评审是合理的，当前仍有现实意义。

现状：

1. 后端采集器已经存在，且 scheduler 也不再是空壳。
2. 但若干数据源仍明显脆弱，例如：
   - `backend/worker/src/services/collect/ustr.ts` 依赖网页结构和多组 CSS 选择器
   - `backend/worker/src/services/collect/yale-budget.ts` 仍写着“示例，实际 URL 需确认”
3. 当前路由层仍可见一些 TODO，说明运维观测与管理接口还不完整。

为什么值得保留：

1. 数据采集类系统最怕“静默失败”，比直接报错更危险。
2. 关税工具的核心信任基础就是数据质量，采集器脆弱性不解决，越往后商业风险越大。

建议保留：

1. 对每个采集器增加“记录数差异阈值”监控，例如与上次相比波动超过一定比例就告警。
2. 增加关键字段非空校验和异常快照。
3. 对第三方整理类或页面抓取类数据源增加更明显的失败可观测性。

---

### 7. 补签月度重置机制不完整

这条是一个小但真实存在的问题。

现状：

1. `src/store/checkin-store.ts` 中定义了 `resetMonthlyMakeup()`。
2. 但当前未看到按月份切换自动触发该逻辑的地方。

为什么值得保留：

1. 这是典型的“规则有定义，生命周期没接上”的问题。
2. 长期运行后会导致补签次数累计错误，影响积分/权益规则的可信性。

建议保留：

1. 在扩展启动、打开主界面或每日定时检查时，对比当前月份与上次重置月份。
2. 若跨月则自动重置 `makeupUsedThisMonth`。

---

### 8. 生产配置尚未环境化

这条建议当前仍然成立。

现状：

1. `src/services/api-client.ts` 的默认 `baseUrl` 仍是 `http://localhost:8000`。
2. `wxt.config.ts` 的 `host_permissions` 当前也只包含 `http://localhost:8000/*`。
3. `PRIVACY.md` 里仍然使用占位邮箱 `privacy@tariff-tool.example.com`。

为什么值得保留：

1. 这类问题不复杂，但都是上线时容易“忘掉却致命”的配置项。
2. 一旦构建产物仍指向 localhost，生产环境就会直接不可用。

建议保留：

1. 用环境变量注入 API 地址和 host permissions。
2. 增加开发/生产配置分离。
3. 上线前替换隐私政策中的占位邮箱与相关对外联系信息。

---

### 9. 钉钉签名实现不正确

这条建议当前仍然完全成立。

现状：

1. `src/services/notification-service.ts` 中的钉钉签名目前只是 `btoa(stringToSign)`。
2. 这不是钉钉要求的 HMAC-SHA256 实现。

为什么值得保留：

1. 如果后续依赖钉钉做采集告警、系统告警或内部通知，这条链路会直接失效。
2. 最危险的是“以为告警发出去了，实际上没有”。

建议保留：

1. 用标准 HMAC-SHA256 方案重写钉钉签名。
2. 同时补一条最小可用的联调验证流程，确认签名真的能被钉钉接受。

---

### 10. 项目缺少最小测试覆盖

这条建议当前仍然非常合理。

现状：

1. `package.json` 里已经配置了 `vitest` 和 `npm test`。
2. 当前项目代码中未见自有的 `*.test.ts` / `*.spec.ts` 文件。

为什么值得保留：

1. 项目已有较复杂的核心逻辑，包括税费计算、积分规则、签到规则、通知规则、支付/权限边界。
2. 这些模块没有最小测试覆盖，后续重构会非常不稳。

建议保留的最小覆盖范围：

1. `strategy-engine`：税率叠加、FTA、De Minimis、排序结果。
2. `credit-engine`：签到奖励、套餐倍数、额度扣减、周限额。
3. `hs-normalize`：各类 HS 编码格式归一化与异常输入。

---

## 三、建议的优先级重排

结合当前仓库状态，建议把仍成立的 Claude 评审点收敛为如下优先级：

### P0：上线前必须解决

1. 真实数据链路闭环：让核心策略计算与通知不再依赖 mock / simulated 数据。
2. 后端权威化权益与计费：前端不再作为最终裁决者。
3. 对 Supabase RLS 和 RPC 权限做逐表、逐身份验证。

### P1：上线前强烈建议

1. `options/App.tsx` 拆分。
2. `detectSimulatedChanges()` 加开发环境守卫或彻底隔离。
3. 修复钉钉签名实现。
4. 环境变量化 API 地址与生产 host permissions。
5. 修复补签月度重置机制。
6. 给采集器加基础数据质量监控。

### P2：尽快补齐

1. 为核心算法与商业化规则补最小测试覆盖。
2. 持续削减 mock 数据在主流程中的占比。
3. 完善采集与通知相关的管理/观测接口。

---

## 四、落地建议

如果只从这份提炼文档里挑最值得马上执行的 5 项，我建议顺序如下：

1. 把 `background.ts` 的模拟通知链路关掉或限制在开发环境。
2. 把积分/试用/套餐的最终判定后移到后端。
3. 跑一轮 Supabase 权限验证清单，确认扩展端没有越权路径。
4. 拆分 `options/App.tsx`，降低后续重构阻力。
5. 补 `strategy-engine`、`credit-engine`、`hs-normalize` 三组最小测试。
